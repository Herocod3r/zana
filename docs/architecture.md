# Design: Zana — Minimal Agent Orchestrator

Generated by /office-hours on 2026-04-05
Branch: main
Repo: Zana
Status: DRAFT
Mode: Builder

## Problem Statement

Managing multiple coding agents (Claude Code, Codex, Aider, etc.) across a project means juggling multiple terminal windows, manually creating git worktrees, losing context when you close tabs, and having no unified view of what's running where. Superset.sh solves this conceptually (projects > worktree workspaces > terminals) but bundles too many features, making it feel verbose, buggy, and slow.

Zana is the anti-superset: a fast, simple desktop app that organizes coding agents into projects and worktree-isolated workspaces with a clean terminal grid. Nothing more.

## What Makes This Cool

Speed and simplicity as a feature. Open Zana, pick a project, create worktrees, spawn agents, watch them work. No config files, no YAML, no daemon, no 20-step setup. It should feel like a native app, not an enterprise tool. The terminal rendering uses ghostty-web (Ghostty's VT parser compiled to WASM), giving you terminal correctness that xterm.js doesn't match, with zero CGo complexity.

## Constraints

- Side project / for fun, with open-source aspirations if it works well
- Must replace the builder's current workflow of managing multiple terminal windows
- Must feel fast and native (not Electron-heavy)
- Must be cross-platform (macOS primary, Linux and Windows secondary)
- No extension system in v1 — but architecture should not preclude one later
- Agent-agnostic: terminals are terminals, no agent-specific logic in core
- Git-only in v1: the entire workspace model is built on git worktrees. Non-git projects are out of scope.
- Single window in v1: no multi-window or multi-monitor support

## Premises

1. **Thin core, extension-ready architecture** — core handles only projects, worktrees, and terminal rendering. The architecture should not preclude a future plugin API, but v1 ships core only with no extension system.
2. **ghostty-web (WASM) for terminals** — drop-in xterm.js-compatible API with Ghostty-correct VT parsing. Runs in the webview. No CGo needed. Fallback to xterm.js if ghostty-web integration hits blockers.
3. **Wails + Go + Vue** — Go backend (PTY management, worktree operations, project persistence), Vue frontend (UI chrome + ghostty-web terminals), Wails for cross-platform packaging with native webview.
4. **Agent-agnostic** — a terminal is a terminal. The core doesn't know or care what agent runs inside it.

## Cross-Model Perspective

Codex (GPT-5.4) reviewed the concept independently. Key insights:

- **Coolest unexplored version:** Treat each workspace as a "durable, replayable unit of thought" — not just code isolation but a record of what an agent did and why. Workspaces become forkable and diffable. This is a future direction, not v1 scope.
- **Revealing quote:** "the best approach is to have the simplest agent orchestrator and allow it to be extensible" — Codex noted the core excitement is building the anti-bloat foundation, not terminals or worktrees per se.
- **50% tool:** tmux gets you halfway (session multiplexing, pane grids, persistence). The other 50% is productization: project model, desktop UX, worktree lifecycle, persistent metadata, session restore.
- **Weekend success criterion:** "Open Zana, select a repo, spin up 3 isolated worktrees, attach Claude/Codex/Aider in parallel, quit, reopen, resume exactly where I was." If that feels instant, you have the product.

## Approaches Considered

### Approach A: Minimal Core (CHOSEN)

Ship the thinnest usable version: projects, worktrees, terminal grid, session persistence. Use it daily for a week before designing the extension API.

- Effort: S (human: ~1 week / CC: ~1-2 days)
- Risk: Low
- Pros: Usable immediately, forces core loop quality, fast feedback
- Cons: No extension system, hardcoded layout, extension architecture deferred

### Approach B: Core + Extension Architecture

Same core but with Go plugin loading, Vue component slots, and event bus from day one. Ship with zero bundled extensions but a real API.

- Effort: M (human: ~2-3 weeks / CC: ~3-5 days)
- Risk: Medium (extension API design without real consumers)
- Pros: Clean separation from day one, community can build early
- Cons: Slower to ship, risk of over-designing plugin API

### Approach C: Workspace-as-Memory

Richer data model where each workspace captures terminal history, agent commands, file changes, and intent notes. Workspaces are forkable, diffable, resumable.

- Effort: M-L (human: ~3-4 weeks / CC: ~5-7 days)
- Risk: Medium-High (complexity before validation)
- Pros: Genuine differentiator, demo-worthy features
- Cons: Risk of building the bloat you're trying to avoid

## Recommended Approach

**Approach A: Minimal Core.** Ship the thinnest thing that replaces the current workflow. Use it for a week. The missing pieces will reveal what the extension API actually needs.

### Key Technical Decisions

1. **PTY data transport:** Wails events (backend push) for terminal output. Wails bindings (frontend call) for terminal input. PTY output bytes are base64-encoded before sending over Wails events (Wails events serialize as JSON; raw bytes must be encoded to survive JSON transport), decoded on the frontend before writing to ghostty-web. **Plan B:** If Wails events cannot sustain 5 MB/s per terminal in the spike (Next Step 2), switch to a localhost WebSocket per workspace before building the rest of the app. The frontend terminal wrapper abstracts the transport, so this swap does not affect Vue components.

1a. **Workspace-worktree mapping:** A workspace is a 1:1 wrapper around a git worktree. Creating a workspace creates a worktree; deleting a workspace removes the worktree (with confirmation). There are no workspaces without worktrees in v1.

2. **Project storage:** SQLite via `modernc.org/sqlite` (pure Go, no CGo). Stores project paths, workspace metadata, terminal layout, session state. Single file per user at `~/.zana/zana.db`.

3. **Worktree management:** Shell out to `git worktree add/list/remove`. No git library needed — the CLI is the stable API.

4. **Terminal grid:** CSS Grid in Vue. New workspaces open with a single terminal pane. Users add panes manually; the grid uses N columns where N = ceil(sqrt(pane_count)), filling rows left-to-right (1 = full, 2 = 1x2, 3 = 2+1, 4 = 2x2, 5 = 3+2, 6 = 2x3). Each workspace has its own terminal grid. The WorkspaceTab bar switches between workspace grids; only one workspace grid is visible at a time. Resize handles allow adjusting pane proportions. Each pane is an independent ghostty-web instance connected to its own PTY.

5. **Terminal input routing:** The focused pane receives all keyboard input. Click-to-focus on panes. Ctrl+Tab cycles focus between panes. Global hotkeys (e.g., Ctrl+Shift+N for new terminal) are captured by the app, not passed to the PTY.

6. **Session restore (visual continuity, not process resume):** On quit, serialize: open project, active workspaces, terminal layout, working directories, AND last 256 KB of rendered output per pane (from the ring buffer). Track working directory via OSC 7 escape sequences (validated and length-capped to prevent injection from PTY content; OSC 7 paths are NOT trusted to set the cwd directly, only to update Zana's cached "last known cwd" for that PTY). On reopen, restore layout, replay the persisted output into each pane (so the user sees their agent's last conversation), and spawn fresh shells in the last known working directories. PTYs are terminated on quit — Zana does NOT resume processes. Logical resume (continuing the agent's conversation) is the agent's responsibility (`claude resume`, `codex --continue`, etc.). Zana's job is preserving the visual context so the user can pick up where they left off.

6a. **Workspace identity:** Workspaces have a display name (human-friendly, auto-generated from a wordlist like city names — swappable later via extension) AND a branch name (proper git ref). These are independent. The New Workspace dialog auto-suggests a display name and lets the user override. The branch name is a separate field. This decouples Zana's mental model ("the Tokyo workspace") from the underlying git ref ("feature/auth-fix").

7. **PTY lifecycle:** PTYs are terminated on app quit. On macOS window close (without quit), PTYs keep running (app stays in dock). On system sleep/wake, PTYs survive (OS handles this). Orphaned PTYs (if the app crashes) are cleaned up on next launch.

8. **Error handling:** If `git worktree add` fails (dirty state, branch conflict, not a git repo), surface the git stderr message in a toast notification. If a PTY fails to spawn, show the error in the terminal pane itself (not a modal). All git operations validate the repo exists and is a git directory before proceeding.

8a. **Git invocation security (CEO 1.2A):** Always invoke git via `exec.Command("git", "worktree", "add", path, "-b", branch)` with separate argv entries. NEVER use `sh -c` or string interpolation. Branch names are validated against git's ref-name rules (no spaces, no shell metacharacters, no leading dash, max 64 chars) before passing to git.

8b. **Path validation:** Project paths are resolved (symlinks followed) and validated as regular directories. Reject paths containing `..`. Reject if the resolved path is outside the user's home directory (configurable later).

8c. **Vue escaping discipline:** `v-html` is forbidden anywhere user-provided strings render. All user input flows through `{{ }}` (auto-escaped). Lint rule recommended.

8d. **Git binary check on startup:** App startup runs `git --version`. If git is not on PATH, show a blocking modal: "Zana requires git to be installed. Install git and relaunch."

8e. **PTY ring buffer + replay (CEO 1.3A):** Each PTY has a 256 KB ring buffer in Go. When a frontend pane mounts (or remounts after tab switch), it requests the buffer via a Wails binding, then subscribes to live events. Closes the race window where Wails events fire before the listener is ready. Buffer contents also feed session persistence (decision 6).

8f. **Workspace creation transaction:** Creating a workspace is a multi-step operation (create worktree, INSERT metadata row, spawn initial PTY). If the SQLite INSERT fails after the worktree was created, ROLLBACK by running `git worktree remove --force` on the just-created worktree. No orphaned worktrees from partial failures.

9. **Worktree cleanup:** Deleting a workspace prompts "Remove worktree and branch?" with options to keep the branch. Force-delete if the worktree has uncommitted changes (with confirmation). On launch, detect worktrees that were manually deleted outside Zana and clean up their metadata.

9a. **New Workspace dialog (CEO 3A):** Three fields: workspace display name (auto-generated from a wordlist, e.g., city names — user can override), branch name (proper git ref, validated), base branch (dropdown of recent branches, default = repo's primary remote HEAD or `main`). Plus a toggle: "create new branch" (default ON) vs "check out existing branch." Detect submodules in the source repo and warn (`git worktree add` does not initialize submodules in the new worktree).

9b. **Closing a pane with running process (CEO 4.1A):** If the PTY's process is still alive on `Wait()`, show a confirm dialog: "<process name> is still running. Kill it?" Default = Cancel. On confirm, send SIGTERM, wait 2s, then SIGKILL.

9c. **Missing project repos (CEO 4.2A):** On launch, if a project's repo path no longer exists (or is no longer a git repo), mark the project as "missing" in the sidebar (greyed out, warning icon). Click → modal with two options: "Re-locate" (file picker) or "Remove from Zana." Workspace metadata is preserved until removal.

9d. **Cross-platform framing:** macOS-first in practice. Linux is a fast follow. Windows is post-v1. Wails handles most cross-platform differences but PTY/shell/OSC 7 behavior varies and v1 doesn't promise Windows.

9e. **Activity indicator dot (CEO accepted scope):** Each terminal pane shows a colored dot driven by last-byte-time per PTY: active (output in last 5s), recent (5-60s), idle (>60s). Tracker lives in `internal/terminal/activity.go`. 1Hz status updates batched into a single Wails event. Stays agent-agnostic.

9f. **Extension seams (A+ approach):** v1 has 3 internal interfaces in Go where extensions will plug in later: `TerminalProvider` (terminal renderer factory), `WorkspaceAction` (right-click menu items), `SidebarPanel` (custom sidebar sections). No extension loader, no plugin system. The seams exist so the v1 core doesn't need a refactor when the extension system arrives in phase 2.

10. **ghostty-web fallback:** The `lib/ghostty.ts` wrapper exposes a minimal interface: `init(container, options)`, `write(data)`, `resize(cols, rows)`, `focus()`, `dispose()`. **Build order:** implement against xterm.js first (known-stable), then integrate ghostty-web behind the same interface. **Go/no-go:** if ghostty-web cannot render a basic interactive shell (vim, htop) correctly in the Wails webview within 2 days of effort, stay on xterm.js for v1.

### Decision E1: Replay handshake with sequence numbers

The CEO 1.3A "ring buffer + replay on mount" had a residual race: Wails events fired between `GetBuffer()` returning and the frontend listener registering would be lost.

Fix:
- Each PTY has a monotonic `seq uint64` counter (starts at 0).
- Each chunk written to the ring buffer is tagged with its seq.
- The ring buffer stores (offset, seq) pairs alongside the bytes so it can find a starting offset for any seq within the buffer's window.
- `pty-out` events include the seq number.
- Frontend mounting flow:
  1. Subscribe to `pty-out:<id>` (queue events but don't apply yet).
  2. Call `GetBuffer(id)` → returns `{data, lastSeq}`.
  3. Apply the buffer data to the terminal.
  4. Drain the queue, dropping events with seq <= lastSeq, applying events with seq > lastSeq.
  5. If there's a gap (next queued event has seq > lastSeq + 1), call `GetBufferSince(id, lastSeq)` to fill the gap.
  6. Apply remaining queued events in order.
- Closes the race fully. No bytes lost.

### Decision E2: OSC 7 is best-effort, not load-bearing

Codex correctly noted that raw zsh/bash do not emit OSC 7 unless shell integration is installed. The previous design treated OSC 7 cwd as a primary mechanism. Fix:

- Session restore spawns each shell in the workspace's **worktree directory** (the path stored in SQLite when the workspace was created). This is the path that's always known and always correct.
- OSC 7 tracking still exists (validated, length-capped, never trusted for shell respawn) but is purely informational. v1 doesn't use it.
- Document in DESIGN.md and architecture.md that "session restore returns you to the workspace's worktree dir, not your last `cd`'d location." Honest framing.
- Remove the "OSC 7" mention from premise 6 in the architecture doc and replace with the simpler approach.

### Decision E5: Process liveness via dedicated wait goroutine

`Wait()` blocks until process exit. It is NOT a liveness check. The previous decision 9b used `Wait()` incorrectly. Fix:

```go
// internal/terminal/pty.go
type PTY struct {
    cmd      *exec.Cmd
    pty      *os.File
    pgid     int           // process group ID for signaling
    done     chan struct{} // closed when Wait() returns
    exitCode int           // set before done is closed
}

func Spawn(cwd, shell string, env []string) (*PTY, error) {
    cmd := exec.Command(shell)
    cmd.Dir = cwd
    cmd.Env = env
    cmd.SysProcAttr = &syscall.SysProcAttr{Setpgid: true} // own process group
    f, err := pty.Start(cmd)
    if err != nil { return nil, err }
    p := &PTY{cmd: cmd, pty: f, pgid: cmd.Process.Pid, done: make(chan struct{})}
    go func() {
        err := cmd.Wait()
        if exitErr, ok := err.(*exec.ExitError); ok {
            p.exitCode = exitErr.ExitCode()
        }
        close(p.done)
    }()
    return p, nil
}

func (p *PTY) IsAlive() bool {
    select {
    case <-p.done: return false
    default:       return true
    }
}

func (p *PTY) Close(force bool) error {
    if !p.IsAlive() { return nil }
    // Signal the process group, not just the shell PID. Catches child processes.
    syscall.Kill(-p.pgid, syscall.SIGTERM)
    select {
    case <-p.done: return nil
    case <-time.After(2 * time.Second):
        if force {
            syscall.Kill(-p.pgid, syscall.SIGKILL)
            <-p.done
            return nil
        }
        return errors.New("process did not exit after SIGTERM")
    }
}
```

### Decision E6: SQLite serialization and busy timeout

Wails-bound methods can run concurrently. modernc/sqlite + database/sql can hit "database is locked" without serialization.

Fix on connection open:
```go
db.Exec("PRAGMA journal_mode=WAL")
db.Exec("PRAGMA busy_timeout=5000")  // 5 seconds
db.Exec("PRAGMA synchronous=NORMAL") // reasonable for WAL
db.SetMaxOpenConns(1)                // serialize all writes through one connection
db.SetMaxIdleConns(1)
db.SetConnMaxLifetime(0)             // no lifetime limit
```

For read-heavy paths post-v1, add a separate read pool. v1 single-conn is fine.

### Decision E7: Workspace creation full transaction

Three steps that can fail: git worktree add, INSERT metadata, spawn initial PTY. Rules:

- **git fails:** Reject. No SQLite write, no PTY. User sees git stderr in toast.
- **INSERT fails after git success:** Rollback. Run `git worktree remove --force <path>`. Log the SQLite error. Return error.
- **PTY spawn fails after git+INSERT success:** **KEEP the workspace.** Log the PTY error. Return the workspace with an empty terminal list. The user can manually spawn a terminal in the workspace later. "No PTY" is a valid workspace state.

### Decision E8: Throughput spike scenarios

The throughput benchmark (Next Step 2) tests these scenarios, NOT just `cat`:
1. `cat` of a 1 MB file (baseline)
2. `npm install` in a real Node project (many small writes, JSON output, slow then fast)
3. `go test ./...` in a Go repo (interleaved test progress)
4. `yes | head -n 100000` (sustained burst)
5. **4 concurrent panes**, each running `go test ./...` simultaneously (the realistic worst case)

Pass criteria:
- All 5 scenarios complete without dropped events
- Single-pane throughput sustained at 5 MB/s minimum (for the burst case)
- No UI stutter (visible 60 fps maintained) during 4-pane concurrent test
- Total memory under 200 MB during the 4-pane test

If any criterion fails: switch to WebSocket sidecar transport before building anything else.

Called from `main.go` BEFORE `wails.Run()`. Failure is non-fatal (log warning, continue with the inherited env).

### Decision E11: Renderer adapter expanded API

The original 5-method interface was too thin. Expanded to ~12 methods to support real terminal needs:

```typescript
// frontend/src/lib/ghostty.ts
export interface TerminalAdapter {
  init(container: HTMLElement, options: TerminalOptions): void
  write(data: string): void                    // base64 input from PTY
  onInput(handler: (data: string) => void): () => void  // returns unsubscribe
  resize(cols: number, rows: number): void
  fit(): { cols: number; rows: number }        // compute size from container
  focus(): void
  blur(): void
  clear(): void
  reset(): void                                // hard reset (escape sequence)
  serialize(): string                          // for session save
  restore(snapshot: string): void              // for session restore
  onTitleChange(handler: (title: string) => void): () => void
  onSelectionChange(handler: (selection: string) => void): () => void
  copySelection(): string
  dispose(): void
}
```

The xterm.js adapter wraps `Terminal` + `FitAddon` + `SerializeAddon`. The ghostty-web adapter implements the same surface. Both are interchangeable.

### Decision E12: Pinia stores never touch reactive state from PTY chunks

PTY data is high-frequency. If the terminal store reacted to every chunk, Vue would re-render the surrounding components on every byte. Fix:

- Each `TerminalPane.vue` holds its terminal instance in a `shallowRef` (or plain non-reactive ref).
- A `useEventListener` wires the Wails `pty-out:<id>` event directly to `terminal.write(data)` in the imperative handler.
- The Pinia `terminal` store only holds metadata (workspace ID, status, last seen seq) — not the data stream.
- Result: no reactivity churn from PTY traffic.

## Open Questions

1. **Wails v2 or v3?** V3 is alpha but has better window APIs. V2 is stable. Need to verify ghostty-web works in Wails' native webview (WebKit on Mac, WebView2 on Windows) — most testing has been in Chromium/Electron contexts.
2. **Terminal output throughput over Wails events.** Need to benchmark: can Wails events push terminal output fast enough for `cat`-ing large files, or do we need a WebSocket sidecar for raw PTY data?
3. **ghostty-web maturity.** The library is young. May hit edge cases. The xterm.js fallback is the safety net.
4. **Session restore fidelity.** How much terminal state (scrollback, cursor position) can we realistically persist and restore? Or is "reopen terminal in the same directory" good enough?
